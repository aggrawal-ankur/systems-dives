# Chunk Description

Each malloc call returns the requested bytes of dynamic memory. Each portion of dynamic memory returned to a process has some bookkeeping attached to it by the allocator. This bookkeeping is kept in a structure called `malloc_chunk`.

***A chunk is a piece of metadata associated with a portion of dynamic memory.*** But when we say "a chunk of memory", it refers to the (chunk_metadata + dynamic_memory).

When the allocator returns the pointer to the dynamic memory carved for the request, we have this metadata chunk sitting right before the payload memory, like this:
```
metadata payload
         ^
         pointer returned to the process
```

The layout of this chunk is as follows:
```c
struct malloc_chunk {

  INTERNAL_SIZE_T       mchunk_prev_size;
  INTERNAL_SIZE_T       mchunk_size;

  struct malloc_chunk*  fd;
  struct malloc_chunk*  bk;

  struct malloc_chunk*  fd_nextsize;
  struct malloc_chunk*  bk_nextsize;
};
```

The authors have acknowledged that this layout is "misleading" in the annotations in malloc.c.
```
/*
  This struct declaration is misleading (but accurate and necessary).
  It declares a "view" into memory allowing access to necessary
  fields at known offsets from a given base. See explanation below.
*/
```
But in my view, the layout is well-reasoned and the lack of reasoning makes it so.

The following is my attempt to make that reasoning visible. It is divided into two parts: 1) explanation of the struct layout, and 2) the actual usage of this layout.

## Chunk Layout Description

The allocator divides allocation sizes as **small** and **large**. Starting from 0, up to a threshold, all the requests are considered small. After that threshold, all the requests are large. Therefore, we have two types of chunks based on **size**.

A chunk can exist in two states: **in-use** and **free**.
  - In-use chunks (both small and large) are self-managed and require nothing beyond the chunk metadata.
  - Free chunks require extra bookkeeping as they are candidates for future malloc requests. Small chunks and large chunks are managed differently.

Based on the information above, the allocator has to manage 3 states of chunks.
  1. **In-use chunks**: chunks the process is actively using (both small and large).
  2. **Small free chunks**: small chunks the process has freed.
  3. **Large free chunks**: large chunks the process has freed.

---

Here is a high level description of how the allocator uses malloc_chunk to represent these 3 states of chunks.

`mchunk_prev_size` stores the size of the previous chunk and `mchunk_size` holds the size of the current chunk.
  - **Note1: size means (chunk_metadata + dynamic_memory).**
  - **Note2: INTERNAL_SIZE_T is a tunable parameter which we will explore in the upcoming sections. For now, consider it `size_t`.**

Free chunks are managed via bins, which are "**circular doubly linked lists**". We have small bins for small chunks and large bins for large chunks.

Small bins manage free chunks of exact size classes, while large bins manage free chunks falling in specific size ranges. For example:
  - we have a small bin of size class 80 bytes. It contains free chunks which have a size of 80 bytes.
  - we have a large bin of size range [1024, 1088) bytes. It manages free chunks of sizes falling in that range. Example - 1024, 1040 and so on.
  - **Note: we will explore the design of these bins in depth in upcoming sections.**

A small bin manage a **linked list** of these free chunks using the `fd/bk` pointers.

A large bin manage two types of links between its free chunks.
  - The `fd/bk` pointers manage links between each free chunk in the bin.
  - Although a large bin manage chunks falling in a specific range, there can be free chunks of same size as well. To link these chunks on top of the generic links, we use the `fd_nextsize/bk_nextsize` pointers.
  - For example, a large bin that manage chunks in this range [1024, 1088) had three free chunks of sizes 1024, 1056, 1024. The three chunks will be linked via fd/bk and the two 1024 size chunks will have an extra fd_nextsize/bk_nextsize link.

Therefore, ***`malloc_chunk` is a generic implementation, designed to provide a single consistent interface for all the three states in which a chunk can exist.*** This is the reasoning behind the layout.

## Usage Description

On 64-bit Linux, both size_t and pointers are 8-bytes wide. Therefore, the size of malloc_chunk is (8*6) 48 bytes. We can verify this with sizeof as well. Just create a c file, copy the definition, create an object and use `sizeof()`.

**Note: For simplicity, we do all the calculations on 64-bit Linux. But the rules are the same for 32-bit Linux. Just use 4 instead of 8.**

Let's discuss our priorities.
  1. mchunk_prev_size and mchunk_size are required in all the cases.
  2. In an in-use chunk, the four pointers aren't useful.
  3. In a small free chunk, fd/bk are required and fd_nextsize/bk_nextsize aren't useful.
  4. In a large free chunk, everything is required.

Based on the priorities above, can we say the following:
  - In an in-use chunk, fd/bk/fd_nextsize/bk_nextsize are garbage, or dereferencing is not defined for them?
  - In a small free chunk, fd_nextsize/bk_nextsize are garbage, or dereferencing is not defined for them?

---

We have two ways to implement this design.
  - **Method1:** Set the required members appropriately and the not required ones NULL.
  - **Method2:** Only set the required members and don't worry about the rest.

In method1, we have to allocate 48 bytes for the metadata, followed by the payload memory. This wastes 24 bytes per in-use chunk, regardless of small or large.

In method2, we have to allocate only 16 bytes for the metadata, followed by the payload memory. This is a better utilization of memory and reduces the overall footprint.

We have two questions to answer.

**Ques1:** "What we will achieve by this meager optimization?". Yes, memory is not a scarce resource today, but that doesn't gives us the right to waste it. Modern operating systems come with huge memory and a high number of processes.
  - If used recklessly, no amount of memory will ever suffice.
  - We have IoT devices and embedded systems, with considerable hardware constraints. Wasting memory is not an option there.

**Ques2:** "Doesn't method2 violates the struct integrity?" It doesn't. Because, we are allocating in heap. We are responsible for managing it. If we dereference a member which is not meant to be dereferenced, we are responsible for the fallout. The allocator knows it, and never does it.

Clearly, method2 is the winner. As someone new to this, the design is not beginner-friendly at all. If you can't understand it in a few minutes, remember, I've invested 4-5 days, reading the annotations and fighting this design.
  - The first step in understanding systems code is to acknowledge the author's design, come with a design you think is better, then contradict it with the author's design.
  - Either you will find a better one, or you will understand the author's design and the constraints that influenced it.

*Therefore, sizeof is not a considerable property of the malloc_chunk struct.*

---

The picture above is almost complete only; one concept remains.

`mchunk_prev_size`, which stores the size of the previous chunk, is a property of the previous chunk's metadata, not the current one. This is probably the most weird and counterintuitive thing about this design.
  - It has a name, "the boundary tag method", where the size of a chunk is managed both in the head and the tail of the chunk.
  - This is the basis of a feature called "coalescing", which allows two consecutive free chunks to get coalesced.

This method is implemented by allocating the (n+1)<sup>th</sup> chunk's metadata 8 bytes before the payload memory of the n<sup>th</sup> chunk ends. We know that it doesn't violate the struct integrity because we are allocating on heap and we are responsible for using the layout correctly.

mchunk_prev_size of the (n+1)<sup>th</sup> chunk is valid only when the n<sup>th</sup> chunk is a free chunk. This answers "what the ending 8 bytes refer to: the payload memory or mchunk_prev_size".

**Note: The first and the last chunks are special, and we will talk about them in the upcoming sections.**

---

That's the reasoning that the annotations lack that makes this design "misleading, but accurate and necessary" in the author's view. The design was never misleading. It lacked the description it deserved.

---

The chunk description above rests on concrete numbers like the width of size_t and the minimum chunk metadata size. These are not hardcoded. They are computed at build time and expressed as macros, making the allocator portable across 32-bit and 64-bit systems without modification."

When the source code is preprocessed, the macro is replaced with the actual value. Take this example:
```c
#include <stdio.h>
#define PI (3.14)

int main(void){
  printf("The area of the circle is given by (pi * r)^2: PI*5*5: %f\n", PI*5*5);
}
```
It becomes:
```c
#include <stdio.h>

int main(void){
  printf("The area of the circle is given by (pi * r)^2: PI*5*5: %f\n", 3.14*5*5);
}
```
... after preprocessing.

But this is not the only use case for macros.

---

Macros can be used to generate named block of codes, which when preprocessed simply replaces the part where the macro is used.

Modern compilers are intelligent due to decades of compiler research. Optimization techniques like **function inlining** were in a complicated state back then.
  - The compilers did support inlining, but it was limited to a single translation unit and not as aggressive as we see it today.
  - Even when C99 formally introduced `inline`, it was still a hint, not a guarantee.

As a result, preprocessing was the only native option for the programmers to achieve similar effects.

Another reason macros are still advantageous (in some scenarios) is that inlining itself is a complex process and preprocessing has ZERO runtime overhead.
  - The compiler has to reason about the code, where it is called and how to inline the code such that it works in harmony.
  - It is not a simple copy-paste. However, preprocessing is exactly a copy-paste routine.

# Allocator's Size Model

Understanding the size model is central to understanding the allocator. It is made up of two things:
  1. Request size
  2. Alignment constraints

Before we explore them, we have to understand the allocator's choice for receiving the request size.

## The use of size_t

malloc() takes size (in bytes) as an argument.

Memory is a limited resource, so there has to be a cap on the maximum dynamic memory that can be requested. That limit is defined by the type in which malloc() receives the size argument.

We need a type which can contain the largest addressable value in an architecture. That type is `size_t`.

***`size_t` is a guarantee from the C standard to be an "unsigned data type" which is exactly as wide as the "pointer type" in an architecture, taking the platform's data model in account. In simple words, size_t is a data type which is wide enough to contain the largest addressable value in an architecture.***
  - On 32-bit Linux, size_t is 4 bytes or 32-bits wide.
  - On 64-bit Linux, size_t is 8 bytes or 64-bits wide.

***`size_t` and the clever use of preprocessing is the basis of an architecture-agnostic implementation.***

---

But size_t is not used directly. It is masked with a type definition.
```c
#define INTERNAL_SIZE_T  size_t
```

This makes `size_t` a tunable parameter.
  - ***A parameter whose value can be tweaked at compile-time is called a tunable parameter (or simply, a tunable).***
  - glibc-malloc has multiple such parameters so that programmers can tweak malloc to their needs.

size_t being a tunable parameter creates a third possibility, where pointers are 8 bytes and INTERNAL_SIZE_T is 4 bytes wide. But this looks counterintuitive to the definition of size_t.
  - We use size_t because we need a type that can hold the largest addressable value in an architecture. That type happens to be the pointer type in that architecture.
  - In certain scenarios, like IoT and embedded systems, the architecture is modern (64-bit) but memory constraints exist. Tweaking INTERNAL_SIZE_T to 4 helps in reducing the memory footprint by reducing the chunk metadata size in an in-use chunk by half.
  - Normally, it is 16 bytes on 64-bit. With INTERNAL_SIZE_T=4, mchunk_prev_size and mchunk_size size 4 bytes each, totaling to 8 bytes.
  - This is not problematic because, we use size_t to store a size, not an address.

---

We know that a struct layout can have padding bytes to satisfy the alignment requirements of struct members, which exist because the hardware requires data to be accessed at addresses that are multiples of a data type's width.
  - When a struct contains different types, the members often require padding after them to keep the next member aligned.
  - This third configuration doesn't create any possibility for padding bytes because each member is already naturally aligned to its own width given the layout order.

The situation is this:
```c
#include <stdio.h>

struct mchunk {
  unsigned int x;        // 4
  unsigned int y;        // 4
  struct mchunk* fd;     // 8
  struct mchunk* bk;     // 8
  struct mchunk* fd2;    // 8
  struct mchunk* bk2;    // 8
};

int main(void){
  struct mchunk x;
  printf("sizeof(x): %d\n", sizeof(x));
}
```
We get 40, which has 0 padding bytes.

---

So the config stack becomes:

| Config # | size_t width | Pointer width |
| :------- | :----------- | :------------ |
| 1 | 4 bytes | 4 bytes |
| 2 | 8 bytes | 8 bytes |
| 3 | 4 bytes | 8 bytes |

Now we are set to explore the size and alignment constraints.

## The model

The whole size model is defined using macros. Before we explore the macros, here is a note.

**Note: The original source code is not optimized for reading. Therefore, I have formatted the source code and all the code snippets quote the formatted source. The logic remains identical.**
  - The original source: [sourceware.org](https://sourceware.org/git/?p=glibc.git;a=blob_plain;f=malloc/arena.c;hb=HEAD)
  - The formatted source: [GitHub.com](https://github.com/aggrawal-ankur/glibc-malloc-explore/blob/main/formatted-malloc.c)

---

Macro #1 is `SIZE_SZ`, which is defined as:
```c
/* The corresponding word size.  */
#define SIZE_SZ  (sizeof(INTERNAL_SIZE_T))
```
It is the width of size_t on the targeted machine's architecture.

| Arch   | SIZE_SZ |
| :---   | :------ |
| 32-bit | 4 bytes |
| 64-bit | 8 bytes |

---

Macro #2 is CHUNK_HDR_SZ, defined as:
```c
/* The chunk header is two SIZE_SZ elements, but this is 
   used widely, so we define it here for clarity later. */
#define CHUNK_HDR_SZ  (2 * SIZE_SZ)
```
It stands for "chunk overhead size", which is the size of metadata in an in-use chunk.

It is twice of size_t to account for mchunk_prev_size and mchunk_size.

| Arch   | CHUNK_HDR_SZ |
| :---   | :----------- |
| 32-bit |  8 bytes |
| 64-bit | 16 bytes |

---

Macro #3 is MIN_CHUNK_SIZE, defined as:
```c
/* The smallest possible chunk */
#define MIN_CHUNK_SIZE    offsetof(struct malloc_chunk, fd_nextsize)
```
It denotes the size of the "smallest possible chunk" on an architecture.

`offsetof` is an ANSI C macro, defined in `stddef.h`, used to determine the byte offset of a specific member from the beginning of its parent structure.

Let's derive manually on 64-bit.
```
  0-7 bytes -> mchunk_prev_size
 8-15 bytes -> mchunk_size
16-23 bytes -> fd
24-31 bytes -> bk
32-39 bytes -> fd_nextsize
40-47 bytes -> bk_nextsize
```
So, MIN_CHUNK_SIZE would be 32 on 64-bit.

| Config # | MIN_CHUNK_SIZE |
| :------- | :------------- |
| 32-bit   | 16 bytes |
| 64-bit   | 32 bytes |
| INTERNAL_SIZE_T=4 | 24 bytes |
---

Macro #4 is MALLOC_ALIGNMENT, defined as:
```c
#define MALLOC_ALIGNMENT  (                   \
  (2 * SIZE_SZ) < __alignof__(long double)    \
	? __alignof__(long double)                  \
  : 2 * SIZE_SZ
)
```
It defines the minimum alignment for malloc'ed chunks.

`__alignof__` is an operator that returns the alignment requirement of a data type (in bytes).

| Arch   | \_\_alignof__(long double) |
| :---   | :------------------------- |
| 32-bit |  4 bytes |
| 64-bit | 16 bytes |

The macro becomes:
```c
// 32-bit (size_t=4)
MALLOC_ALIGNMENT == (8 < 4)  ?  4  :  8  == 8

// 64-bit (size_t=8)
MALLOC_ALIGNMENT == (16 < 8)  ?  16  :  16  == 16
```

In both the cases, the alignment is kept twice of size_t's width. This is because malloc is a general purpose allocator.
  - It is unaware of what the caller will store in the returned memory.
  - Therefore, it ensures that the returned memory is aligned to all the fundamental types the C standard provides. This happens to be `double` on 32-bit and `long double` on 64-bit.

However, it's real importance is in that third configuration, where pointers are 8 bytes and INTERNAL_SIZE_T is 4 bytes wide.
```c
MALLOC_ALIGNMENT = (2 * 4) < 16  ?  16  :  16
```

Notice how the alignment is kept 16 bytes even when INTERNAL_SIZE_T is 4 bytes. This further strengthens our reasoning behind the third case. It does two things explicitly:
  1. reducing the metadata overhead per chunk by 8 bytes, and
  2. drastically reducing the cap on the maximum request size.

The tradeoffs are acceptable on an IoT device.

| Config # | MALLOC_ALIGNMENT |
| :------- | :--------------- |
| 32-bit   |  8 bytes |
| 64-bit   | 16 bytes |
| INTERNAL_SIZE_T=4 | 16 bytes |

---

Macro #5 is MALLOC_ALIGN_MASK, defined as:
```c
/* The corresponding bit mask value. */
#define MALLOC_ALIGN_MASK  (MALLOC_ALIGNMENT - 1)
```

MALLOC_ALIGNMENT is a power-of-2 value and MALLOC_ALIGN_MASK is the bit mask of MALLOC_ALIGNMENT. MALLOC_ALIGN_MASK has those lower bits set which are clear in MALLOC_ALIGNMENT.
```bash
# 64-bit
MALLOC_ALIGNMENT  = 16 = 0001 0000
MALLOC_ALIGN_MASK = 15 = 0000 1111
```

It is used in a variety of bitwise operations.
  1. Check if a size/address is aligned to the alignment boundary.
     ```c
     (addr & MALLOC_ALIGN_MASK) == 0  :=  aligned
     (addr & MALLOC_ALIGN_MASK) != 0  :=  misaligned
     ```
  2. Round a size/address up to the alignment boundary.
     ```c
     -> (size + MALLOC_ALIGN_MASK) & ~MALLOC_ALIGN_MASK
     -> (33 + 15) & ~15
     -> 48 & -16
     -> 48
     ```
  3. Round a size/address down to the alignment boundary.
     ```c
     -> size & ~MALLOC_ALIGN_MASK
     -> 41 & ~15
     -> 32
     ```

| Config # | MALLOC_ALIGN_MASK |
| :------- | :---------------- |
| 32-bit   |  7 |
| 64-bit   | 15 |
| INTERNAL_SIZE_T=4 | 15 |

---

Macro # is MINSIZE, defined as:
```c
/* The smallest size we can malloc is an aligned minimal chunk */
#define MINSIZE    (unsigned long)( \
  ( (MIN_CHUNK_SIZE+MALLOC_ALIGN_MASK) & ~MALLOC_ALIGN_MASK)
)
```

| Config # | MINSIZE |
| :------- | :------ |
| 32-bit   | 16 bytes |
| 64-bit   | 32 bytes |
| INTERNAL_SIZE_T=4 | 32 bytes |
