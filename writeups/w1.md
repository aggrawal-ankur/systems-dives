glibc-malloc is a userspace allocator. It requests virtual memory from the kernel and releases it in chunks to the process when it requests. When that memory is actually used, the kernel backs it with physical memory. This is true for almost everything because virtual memory is the interface we deal with.

# Section 1: Chunk Description

Each malloc call returns the requested bytes of dynamic memory. Each portion of dynamic memory returned to a process has some bookkeeping attached to it by the allocator. This bookkeeping is kept in a structure called `malloc_chunk`.

***A chunk is a piece of metadata associated with a portion of dynamic memory.*** But when we say "a chunk of memory", it refers to the (chunk_metadata + dynamic_memory).

When the allocator returns the pointer to the dynamic memory carved for a request, we have this metadata chunk sitting right before the payload memory, like this:
```
metadata payload
         ^
         pointer returned to the process
```

The layout of this chunk is as follows:
```c
struct malloc_chunk {

  INTERNAL_SIZE_T       mchunk_prev_size;
  INTERNAL_SIZE_T       mchunk_size;

  struct malloc_chunk*  fd;
  struct malloc_chunk*  bk;

  struct malloc_chunk*  fd_nextsize;
  struct malloc_chunk*  bk_nextsize;
};
```

The authors have acknowledged that this layout is "misleading" in the annotations in malloc.c.
```
/*
  This struct declaration is misleading (but accurate and necessary).
  It declares a "view" into memory allowing access to necessary
  fields at known offsets from a given base. See explanation below.
*/
```
But in my view, the layout is well-reasoned and the lack of reasoning makes it so.

The following is my attempt to make that reasoning visible. It is divided into two parts:
  1. the explanation of the struct layout, and
  2. the actual usage of this layout.

## Layout Description

The allocator divides allocation sizes as **small** and **large**. Starting from 0, up to a threshold, all the requests are **small**. Requests that cross that threshold are **large**.

Therefore, we have two types of chunks based on **size**: small chunks and large chunks.

---

A chunk can exist in two states: **in-use** and **free**.
  - **In-use chunks** (both small and large) are self-managed and require nothing beyond the usual chunk metadata.
  - **Free chunks** require extra bookkeeping as they can be reused in servicing future malloc requests.

**Note: Small free chunks and large free chunks are managed differently.**

---

Based on the information above, the allocator has 3 chunk states to manage.
  1. **In-use chunks**: chunks the process is actively using (both small and large).
  2. **Small free chunks**: small chunks the process has freed.
  3. **Large free chunks**: large chunks the process has freed.

Here is a high level description of how malloc_chunk is used to represent these 3 states of chunks.

`mchunk_prev_size` stores the size of the previous chunk and `mchunk_size` holds the size of the current chunk.
  - **Note1: size means (chunk_metadata + dynamic_memory).**
  - **Note2: INTERNAL_SIZE_T is a tunable parameter which we will explore in the upcoming sections. For now, consider it `size_t`.**
  - The reason we store the size of the (n-1)<sup>th</sup> chunk in the n<sup>th</sup> chunk is out-of-scope for this section and is explained in section3.

---

Free chunks are managed via bins, which are "**circular doubly linked lists**". We have small bins for small chunks and large bins for large chunks.

Small bins manage free chunks of exact size classes, while large bins manage free chunks falling in specific size ranges. For example, we have a:
  - small bin of size class 80 bytes. It contains free chunks sized 80 bytes.
  - large bin of size range [1024, 1088) bytes. It contains free chunks of sizes falling in that range. Example - 1024, 1040 and so on.
  - **Note: we will explore the design of these bins in depth in upcoming sections.**

---

A small bin manage a **linked list** of free chunks using the `fd/bk` pointers.

A large bin manage two types of links between its free chunks.
  - The `fd/bk` pointers manage links between each free chunk in the bin.
  - Although a large bin manage chunks falling in a specific size range, there can be free chunks of same size as well. To link these chunks on top of the generic links, we use the `fd_nextsize/bk_nextsize` pointers.
  - For example, a large bin managing chunks in this range [1024, 1088) had three free chunks of sizes 1024, 1056, 1024. The three chunks will be linked via fd/bk and the two 1024 size chunks will have an extra fd_nextsize/bk_nextsize link.

---

Therefore, ***malloc_chunk is a generic implementation, designed to provide a single consistent interface for all the three states in which a chunk can exist.*** This is the reasoning behind the layout. Let's discuss how the layout is actually used.

## Usage Description

**Note: For simplicity, we do all the calculations on 64-bit Linux. But the rules are the same for 32-bit Linux. Just use 4 instead of 8.**

On 64-bit Linux, both size_t and pointers are 8-bytes wide. That means, the size of malloc_chunk is (8*6) 48 bytes. We can verify this with sizeof as well. Just create a c file, copy the definition, create an instance and use `sizeof()` on it.

In the previous section, we have understood that malloc_chunk is a generic implementation. The consequence of this is that each of the 3 states are represented differently.
  1. mchunk_prev_size and mchunk_size are necessary in all the cases.
  2. In an in-use chunk, the four pointers aren't useful.
  3. In a small free chunk, fd/bk are required and fd_nextsize/bk_nextsize aren't useful.
  4. In a large free chunk, everything is required.

Basically,
  - in an in-use chunk, fd/bk/fd_nextsize/bk_nextsize are garbage, or, dereferencing is not defined for them?
  - In a small free chunk, fd_nextsize/bk_nextsize are garbage, or, dereferencing is not defined for them?

---

We have two ways to implement this design.
  - **Method1:** Set the required members appropriately and the not required ones NULL.
  - **Method2:** Only set the required members and don't worry about the rest.

Let's calculate how these methods perform for an in-use chunk.
  - In method1, we have to allocate full 48 bytes for the metadata, followed by the payload memory. This wastes 24 bytes per in-use chunk, regardless of small or large.
  - In method2, only the initial 16 bytes in the metadata struct are usable, followed by the payload memory. This is a better utilization of memory and reduces the overall footprint.

Here is an illustration to visualize method2.
```
-----------------------------------------------------------------
| prev_size | mchunk_size | fd | bk | fd_nextsize | bk_nextsize |
-----------------------------------------------------------------
                            ^
                            Dynamic memory starts here
```
  - Method2 prevents the trailing 24 bytes in the struct from getting wasted.
  - Those members still exist. We can do `struct->fd`, but as decided, they aren't useful in an in-use chunk, thus kept garbage by utilizing that area of memory as dynamic memory.

---

Clearly, method2 is the winner.

As someone new to this, the design is not beginner-friendly at all. If you can't understand it in the first attempt, remember this, I've invested 4-5 days, reading the annotations and fighting this design. And the description you have just read is a result of multiple rewrites.

Either this act will make the author's design clear, or you'll end up finding a better one. Either way, it's a win.

The annotations lacked this reasoning, which made this design look "misleading, but accurate and necessary" in the author's view. It was never misleading.

The description above rests on some concrete numbers like "the width of size_t" and "the minimum chunk metadata size". These are not hardcoded. They are computed at build time and expressed as macros, making the allocator portable across different architectures."

When the source code is preprocessed, the macro is replaced with the actual value. Take this example:
```c
#include <stdio.h>
#define PI (3.14)

int main(void){
  printf("The area of the circle is given by (pi * r)^2: PI*5*5: %f\n", PI*5*5);
}
```
It becomes:
```c
#include <stdio.h>

int main(void){
  printf("The area of the circle is given by (pi * r)^2: PI*5*5: %f\n", 3.14*5*5);
}
```
... after preprocessing.

But this is not the only use case for macros.

---

Macros can be used to generate named block of codes, which when preprocessed simply replaces the part where the macro is used.

Modern compilers are intelligent due to decades of compiler research. But optimization techniques like **function inlining** were in a complicated state in the early days.
  - The compilers did support inlining, but it was limited to a single translation unit and not as aggressive as we see it today.
  - Even when C99 formally introduced `inline`, it was still a hint, not a guarantee.

As a result, preprocessing was the only native option for the programmers to achieve similar effects.

Another reason macros are still advantageous (in some scenarios) is that inlining is not a simple copy-paste. 
  - The compiler has to understand the overall situation to inline a function such that everything works in harmony.
  - On the other hand, preprocessing is a simple copy-paste routine with ZERO runtime overhead.

The chunk description above makes some solid claims about the layout of malloc_chunk. We can verify these claims further using `chunk2mem(p)`.

It is defined as:
```c
#define chunk2mem(p)    ( (void*) ((char*)(p) + CHUNK_HDR_SZ) )
```

`chunk2mem` takes a pointer to a chunk, casts it to `char*` (for pointer arithmetic) and add "chunk header size" to it.
  - Chunk header is the size of metadata bytes in an in-use chunk, which are (2 * SIZE_SZ) bytes. We will discuss CHUNK_HDR_SZ in the next section in detail.
  - This will land us at the `fd` member in the struct, where the payload memory starts, exactly as discussed.

# Section 2: Allocator's Size Model

Understanding the size model is central to understanding the allocator. It consists of two things:
  1. Request size.
  2. Alignment constraints.

Before we explore them, we have to understand the allocator's choice for receiving the request size.

## The use of size_t

malloc() takes a size (in bytes) as its argument.

Each data type has a maximum addressable limit. We need a type which can contain the largest addressable value in an architecture. The answer is `size_t`.

***`size_t` is a guarantee from the C standard to be an "unsigned integer data type", which is exactly as wide as the "pointer type" in an architecture, taking the platform's data model in account. In simple words, size_t is a data type which is wide enough to contain the largest addressable value in an architecture.***

| Arch | size_t |
| :--- | :----- |
| 32-bit Linux | 4 bytes |
| 64-bit Linux | 8 bytes |

***`size_t` and the clever use of preprocessing is the basis of an architecture-agnostic implementation.***

---

But size_t is not used directly. It is masked with a type definition.
```c
#define INTERNAL_SIZE_T  size_t
```
This makes `size_t` a tunable parameter.

***A parameter whose value can be tweaked at compile-time is called a tunable parameter (or, a tunable).*** There are multiple such parameters provided for the programmers to customize malloc to their needs.

size_t being a tunable creates a third possibility, where pointers are 8 bytes and INTERNAL_SIZE_T is 4 bytes wide. But this looks counterintuitive to the definition of size_t.

In certain IoT and embedded system configurations, the architecture is modern (64-bit) with memory constraints. Tweaking INTERNAL_SIZE_T to 4 does two things.
  1. The metadata size per in-use chunk is shrunk by half, reducing the overall memory footprint. With INTERNAL_SIZE_T=4, mchunk_prev_size and mchunk_size size 4 bytes each, totaling to 8 bytes.
  2. The maximum request size is reduced drastically to ~4 GiB of virtual memory.

Tuning INTERNAL_SIZE_T is not problematic because, we use size_t to store a size, not an address. The tradeoff is also explicit and acceptable provided the constraints.

---

A struct with members of different types often require padding bytes to keep access within the struct aligned.
  - It exist because the hardware requires data to be accessed at addresses that are multiples of a data type's width.
  - INTERNAL_SIZE_T=4 doesn't create possibility for padding bytes as each member is naturally aligned to its own width given the layout order.

This is the situation:
```c
#include <stdio.h>

struct mchunk {
  unsigned int x;        // 4
  unsigned int y;        // 4
  struct mchunk* fd;     // 8
  struct mchunk* bk;     // 8
  struct mchunk* fd2;    // 8
  struct mchunk* bk2;    // 8
};

int main(void){
  struct mchunk x;
  printf("sizeof(x): %d\n", sizeof(x));
}
```
We get 40, which has 0 padding bytes.

---

To summarize, there are the three configurations the allocator must handle.

| Config # | size_t width | Pointer width |
| :------- | :----------- | :------------ |
| 1 | 4 bytes | 4 bytes |
| 2 | 8 bytes | 8 bytes |
| 3 | 4 bytes | 8 bytes |

---

Now we are set to explore the size and alignment constraints.

## The model

The whole size model is defined using macros. Before we explore these macros, here is a note.

**Note: The original source code is always receiving changes by contributors and maintainers. As a result, it is not formatted for readability. Therefore, I have formatted the source code and all the code snippets quote the formatted source. The logic remains untouched.**
  - The original source: [sourceware.org](https://sourceware.org/git/?p=glibc.git;a=blob_plain;f=malloc/arena.c;hb=HEAD)
  - The formatted source: [GitHub.com](https://github.com/aggrawal-ankur/glibc-malloc-explore/blob/main/formatted-malloc.c)

---

### Macro #1 -> `SIZE_SZ`

It is the width of size_t on the targeted machine's architecture.
```c
/* The corresponding word size. */
#define SIZE_SZ  (sizeof(INTERNAL_SIZE_T))
```

| Arch   | SIZE_SZ |
| :---   | :------ |
| 32-bit | 4 bytes |
| 64-bit | 8 bytes |

### Macro #2 -> CHUNK_HDR_SZ

It stands for "chunk header size", which refers to the metadata that sits at the beginning of an in-use chunk i.e, "mchunk_prev_size and mchunk_size".
```c
#define CHUNK_HDR_SZ    (2 * SIZE_SZ)
```

| Arch   | CHUNK_HDR_SZ |
| :---   | :----------- |
| 32-bit | 8 bytes |
| 64-bit | 16 bytes |

### Macro #3 -> MIN_CHUNK_SIZE

It denotes the size of "the smallest possible chunk" in an architecture.
```c
#define MIN_CHUNK_SIZE    offsetof(struct malloc_chunk, fd_nextsize)
```

`offsetof` is an ANSI C macro, defined in `stddef.h`, used to determine the byte offset of a specific member from the beginning of its parent structure.

Let's derive it manually on 64-bit.
```
  0-7 bytes -> mchunk_prev_size
 8-15 bytes -> mchunk_size
16-23 bytes -> fd
24-31 bytes -> bk
32-39 bytes -> fd_nextsize
40-47 bytes -> bk_nextsize
```
So, MIN_CHUNK_SIZE would be 32 on 64-bit.

| Config # | MIN_CHUNK_SIZE |
| :------- | :------------- |
| 32-bit   | 16 bytes |
| 64-bit   | 32 bytes |
| INTERNAL_SIZE_T=4 | 24 bytes |

### Macro #4 -> MALLOC_ALIGNMENT

It defines the minimum alignment for in-use chunks.
```c
#define MALLOC_ALIGNMENT  (                   \
  (2 * SIZE_SZ) < __alignof__(long double)    \
	? __alignof__(long double)                  \
  : 2 * SIZE_SZ
)
```

`__alignof__` is an operator that returns the alignment requirement of a data type (in bytes).

| Arch   | \_\_alignof__(long double) |
| :---   | :------------------------- |
| 32-bit |  4 bytes |
| 64-bit | 16 bytes |

The macro becomes:
```c
// 32-bit (size_t=4)
MALLOC_ALIGNMENT == (8 < 4)  ?  4  :  8  == 8

// 64-bit (size_t=8)
MALLOC_ALIGNMENT == (16 < 8)  ?  16  :  16  == 16
```

In both the cases, the alignment is kept twice of size_t's width. This is because malloc is a general purpose allocator. It is unaware of what the caller will store in the returned memory.
  - So it ensures that the returned memory is aligned to all the fundamental types the C standard provides.
  - The largest fundamental type in 32-bit is `double`, which has a width of 8 bytes and `long double` in 64-bit, which has a width of 16 bytes. This is what MALLOC_ALIGNMENT is.

However, it's real importance is in that third configuration, where pointers are 8 bytes and INTERNAL_SIZE_T is 4 bytes wide.
```c
MALLOC_ALIGNMENT = (2 * 4) < 16  ?  16  :  16
```

Notice how the alignment is decided based on the largest fundamental type, not the value of INTERNAL_SIZE_T.

---

| Config # | MALLOC_ALIGNMENT |
| :------- | :--------------- |
| 32-bit   |  8 bytes |
| 64-bit   | 16 bytes |
| INTERNAL_SIZE_T=4 | 16 bytes |

### Macro #5 -> MALLOC_ALIGN_MASK

MALLOC_ALIGNMENT is a power-of-2 value and MALLOC_ALIGN_MASK is the bit mask of it.
```c
#define MALLOC_ALIGN_MASK    (MALLOC_ALIGNMENT - 1)
```

MALLOC_ALIGN_MASK has the lowest 4 bits set which are clear in MALLOC_ALIGNMENT.
```bash
# 64-bit
MALLOC_ALIGNMENT  = 16 = 0001 0000
MALLOC_ALIGN_MASK = 15 = 0000 1111
```

It is used in a variety of bitwise operations.
  1. Check if a size/address is aligned to the alignment boundary (the lower 4-bits in the addr/size must be all 0 to yield a zero against all 1s of the bit-mask).
     ```c
     (addr & MALLOC_ALIGN_MASK) == 0  :=  aligned
     (addr & MALLOC_ALIGN_MASK) != 0  :=  misaligned
     ```
  2. Round a size/address up to the alignment boundary.
     ```c
     -> (size + MALLOC_ALIGN_MASK) & ~MALLOC_ALIGN_MASK
     -> (34 + 15) & ~15
     -> 49 & -16
     -> 48
     ```
  3. Round a size/address down to the alignment boundary.
     ```c
     -> size & ~MALLOC_ALIGN_MASK
     -> 41 & ~15
     -> 32
     ```
---

| Config # | MALLOC_ALIGN_MASK |
| :------- | :---------------- |
| 32-bit   |  7 |
| 64-bit   | 15 |
| INTERNAL_SIZE_T=4 | 15 |

### Macro #6 -> MINSIZE

It is the smallest size that malloc supports.
```c
#define MINSIZE    (unsigned long)( \
  ( (MIN_CHUNK_SIZE + MALLOC_ALIGN_MASK) & ~MALLOC_ALIGN_MASK)
)
```

We know that an in-use chunk requires (2 * SIZE_SZ) bytes for storing metadata and when it is freed, it requires (2 * ptr_width) bytes to manage fd/bk. That totals to 16 bytes on 32-bit and 32 bytes on 64-bit.

In INTERNAL_SIZE_T=4, the metadata overhead is 24 bytes, but 24 is not aligned to the alignment boundary (16-div), so we round it up to the next alignment boundary, making MINSIZE 32 bytes.

| Config # | MINSIZE |
| :------- | :------ |
| 32-bit   | 16 bytes |
| 64-bit   | 32 bytes |
| INTERNAL_SIZE_T=4 | 32 bytes |

---

If you notice, MIN_CHUNK_SIZE and MINSIZE look identical. They vary only in the third config. However, they are different.
  - MIN_CHUNK_SIZE defines the "smallest possible chunk size" **structurally**. It doesn't take alignment constraints in account.
  - MINSIZE is the actual policy implemented by the allocator. It takes the minimum size possible structurally and add alignment constraints to it. The values happen to be equal in the first two configurations because the struct aligned with the alignment constraints. That broke with INTERNAL_SIZE_T=4.

The most easiest way to verify this is by searching for the occurrences of both in malloc.c. You will find that MIN_CHUNK_SIZE is mentioned only two times, first in the definition and second in calculating MINSIZE. However, MINSIZE has 55 occurrences.

---

To summarize, the size model describes a single chunk in isolation. However, a standard Linux process calls malloc and free several times. This repeated allocation-deallocation fragments the memory.

The fragmented memory can exist in two layouts, depending on the malloc-free sequence.
  1. {...., in-use, free, in-use, free, ....}
  2. {...., in-use, free, free, in-use, ....}

Fragmentation increases the pressure on physical memory because the freed chunks are still backed by physical memory but not utilized by the process.
  - The only way to reduce this pressure is to reallocate the freed chunks.
  - This reallocation completely depends on the process asking for a size which when aligned is available as a free chunk.

Furthermore, the second layout creates an amplified version of this problem.
  - Suppose two adjacent chunks of sizes 48 bytes and 48 bytes were freed. The next malloc request asked for an aligned 96 bytes.
  - Can we reuse these 96 bytes? NO. The memory is contiguous yet fragmented across two chunks.

Layout1 creates a fragmentation the allocator can't do anything about. But layout2 creates a fragmentation which the allocator can absolutely manage to some extent. Take this:
  - The probability of the process asking for another aligned 48 bytes bytes chunk might be less, but the probability of asking an aligned size which falls in the range of 48-96 bytes is definitely higher.
  - But this is possible only when the adjacent free chunks are coalesced and the memory always exist in layout1.

To implement coalescing, we need two things.
  1. A way to identify if the next/prev chunk is free.
  2. If the next/prev chunk is free, we need a way to reach that chunk from the current chunk.

A computer scientist and mathematician named **Donald Knuth** has found a way to embed coalescing support directly in the chunk metadata. The strategy is discussed in his book *The Art Of Computer Programming, Volume 1: Fundamental Algorithms*, paragraph 4, page 440. It is called, **the boundary tag method**.

The authors of this allocator use the same strategy and the rest of the chunk description is about that.
